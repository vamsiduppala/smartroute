# Classifier calibration (credit-free — no model calls)

How the zero-model-call `ComplexityClassifier` picks a **starting** tier, measured
against a hand-labeled corpus (`eval/classifier-cases.jsonl`). Generated by
`ClassifierCalibrationTest` — do not hand-edit.

The heuristic only sees a proxy for difficulty (length, code fences, hard-signal
keywords), so it diverges from intent. Direction is what matters:
**undershoot** is safe (the router escalates on a failed check — one extra cheap
attempt), **overshoot** is not (there is no de-escalation, so it overspends every
time). The heuristic is tuned to prefer undershoot.

**Reading this:** overshoot is 0 by design — the classifier never starts above the
intended tier on this corpus, so it never overspends. The undershoots are the
heuristic being conservative (e.g. a proof phrased with "deriving" rather than
"derive", or a lightly-worded trade-off question); escalation catches those at the
cost of one cheap first attempt. This is the intended trade: a cheap, explainable
guess that errs toward the cheaper tier and lets validation-driven escalation
recover the misses. See [simulation-results.md](simulation-results.md) for what that
costs in aggregate, and [routing-NOTES.md](routing-NOTES.md) for the mechanism.
