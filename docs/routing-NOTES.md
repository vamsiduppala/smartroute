# routing module — build notes

**Rides:** GPT-5.6 (Sol/Terra/Luna), launched 2026-07-09 — this is the core module the rest of the gateway wraps.

**What it does:**
- `ComplexityClassifier` — cheap, zero-model-call heuristic (length + regex hard-signals + code-fence detection) that picks a **starting** tier. It only needs to be roughly right, because escalation catches the misses.
- `SmartRouteService` — walks the tiers Luna → Terra → Sol, calling the model at each rung and checking the answer with a `Validator`, until one passes or Sol also fails. `route(prompt, validator)` uses the classifier's pick as the start; `routeFrom(prompt, validator, startTier)` forces a specific start tier instead (used by the gateway's governance layer — see `gateway-NOTES.md`). Each real model call within the escalation is captured as an `AttemptRecord` (tier, tokens, cost, and latency measured directly around that one call) — `RouteResult.attemptRecords()` carries the full per-attempt breakdown alongside the pre-existing aggregate totals.

**The gotcha that matters most (see BUILD_NOTES.md):** cost is billed **per attempt at that attempt's own rate**, not at the final tier's rate. A Luna attempt that fails and escalates to Terra costs Luna-rate + Terra-rate, not just Terra-rate. Routing is not free — a classifier that under-shoots on a hard task can cost *more* than going straight to Sol.

**Gotcha #1 (resolved 2026-07-13):** that per-attempt cost was only tracked when every attempt in the loop actually got a response. If Luna succeeded (real cost incurred, answer rejected) and Terra's call itself then threw — network error, upstream 5xx, timeout — the bare exception propagated out of the loop and Luna's already-real cost vanished: never reached `RouteResult`, so never reached `SpendLedger` or `TelemetryService` either. Fixed with `PartialRouteException`, which carries a `RouteResult` snapshot of whatever was accumulated before the failing attempt (only earlier, *successful* attempts contribute cost — the attempt that actually failed contributes zero, since no tokens came back). Callers that care about that partial cost catch it, record/book it, then rethrow so the failure still surfaces the same way it always did. See `gateway-NOTES.md` for the ledger side and `observability-NOTES.md` for the telemetry side.

**Gotcha #2 (resolved 2026-07-13, same day, found reviewing gotcha #1's own fix):** the try/catch that produces `PartialRouteException` only wrapped `chatModel.call(...)` itself — but a call can SUCCEED and still return a malformed response (no generations) that throws while being parsed, one line later. That parsing exception propagated bare, reopening exactly the cost-loss gap gotcha #1 just closed. Fixed by moving the whole attempt (call + parse) inside one try block. Confirmed with a test that constructs a real `ChatResponse` with zero generations (a genuine NPE from Spring AI's own `getResult()`, not a mocked failure) after an earlier successful attempt.

**Endpoints:** `POST /route` (routing alone, no guardrails/governance wrapping — see `GatewayController` / `POST /gateway/route` for the full pass).
**Tests:** `SmartRouteServiceTest` — pass-at-start-tier, escalate-one-tier (with the per-attempt-cost assertion), climb-to-Sol-without-looping-forever, `routeFrom` overriding the classifier, and three `PartialRouteException` cases (failure after an earlier success preserves that cost; failure on the first attempt reports zero; a malformed-but-non-throwing response after an earlier success). `ComplexityClassifierTest` covers the tier-selection heuristic. `RouterTelemetryAspectTest` (see `observability-NOTES.md`) covers per-attempt telemetry recording via `AttemptRecord`. All offline: `OpenAiChatModel` is mocked, `ChatResponse`/`Generation`/`AssistantMessage` are constructed as real objects (their accessors are `final` — Mockito can't stub them).
